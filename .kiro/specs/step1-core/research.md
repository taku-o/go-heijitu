## 4. 設計フェーズに持ち込むべき検討事項

### A. `WithConfig` のシグネチャ問題（要設計決定）

設計書では `func WithConfig(configPath string) (Option, error)` となっている。
Functional Options パターンの標準的な使い方は `func WithXxx(...) Option` だが、
`WithConfig` はファイル読み込みを伴うためエラーが発生しうる。

選択肢：
- **案1**: `func WithConfig(configPath string) (Option, error)` — 呼び出し側が `if err != nil` で処理する（設計書通り）
- **案2**: `func WithConfig(configPath string) Option` — エラーを Option 内部に保持し、`New()` 呼び出し時にエラーを表面化させる

設計書では案1を採用しているため、案1で進める。設計フェーズで確認すること。

### B. テスト用モックプロバイダーの置き場所

`IsBusinessDay` のテストにはモックプロバイダーが必要。
Go の慣例として `_test.go` ファイル内にローカル定義するか、`internal/testutil` パッケージを作るかを設計で決める。
Step 1 の範囲ではシンプルに `calendar_test.go` 内にローカル定義する方針を推奨する。

### C. `Config` 型のパッケージ内公開範囲

`Config` 型は `WithConfig` の内部処理で使用するが、ライブラリ利用者が直接扱う必要はない。
設計フェーズで公開型（`Config`）にするか非公開型（`config`）にするかを決める。

## 7. プロジェクト横断ルール（設計・実装への制約）

エラーハンドリングについてプロジェクト全体で以下のルールが適用される。実装・設計時に必ず守ること。

- **エラーは即座に呼び出し元へ返す。** 内部で握りつぶしてはならない。
- **フォールバック処理は実装しない。** エラー発生時の代替動作（デフォルト値返却・リトライ等）は一切不要。
- すべての公開 API において、エラーが発生した場合は `error` を返す設計とする。
